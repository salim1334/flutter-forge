# Publishing Flutter Forge to npm (professional checklist)

Use this before `npm publish`. Check each item once done.

---

## 1. Package identity

- [ ] **Name available** ‚Äî search [npmjs.com](https://www.npmjs.com/search?q=create-flutter-forge)
- [ ] **package.json** fields filled:
  - `name`: `create-flutter-forge`
  - `version`: start at `1.0.0` (semver)
  - `description`: clear one-liner
  - `keywords`: flutter, getx, cli, scaffold, starter
  - `author`: your name + email (or `"Your Name <you@email.com>"`)
  - `license`: MIT (matches `LICENSE` file)
  - `repository`, `homepage`, `bugs` (GitHub URLs)
- [ ] **README** has install command, prerequisites, and link to `docs/`

---

## 2. Legal & trust

- [ ] `LICENSE` file (MIT) with correct copyright year and name
- [ ] No secrets in repo (`.env`, API keys, keystores)
- [ ] `.gitignore` covers `node_modules/`, `dist/`, test projects, `.env`
- [ ] Optional: `SECURITY.md` for vulnerability reporting
- [ ] Optional: `CODE_OF_CONDUCT.md` if you want community contributions

---

## 3. Code quality

- [ ] `npm run build` succeeds (TypeScript ‚Üí `dist/`)
- [ ] `npm pack --dry-run` lists only intended files (`bin`, `dist`, `templates`)
- [ ] Local smoke test:

  ```bash
  node bin/create-flutter-forge.js test_app --yes --platforms android
  cd test_app && flutter analyze --no-fatal-infos && flutter test
  npx flutter-forge feature cart --with-model
  ```

- [ ] GitHub Actions CI green (`.github/workflows/ci.yml`)
- [ ] `CHANGELOG.md` updated for each release

---

## 4. npm account & publish

```bash
npm login
npm whoami
npm publish --access public   # first time for unscoped package
```

- [ ] Use **2FA** on npm account (recommended: authorization-only publish)
- [ ] Tag release in git: `git tag v1.0.0 && git push origin v1.0.0`
- [ ] After publish, verify install from registry (not local link):

  ```bash
  cd /tmp
  npm create flutter-forge@latest verify_app -- --yes
  ```

---

## 5. Versioning (semver)

| Change | Bump |
|--------|------|
| Breaking template or CLI behavior | **major** (2.0.0) |
| New feature subcommand option, new template files | **minor** (1.1.0) |
| Bug fixes, docs only | **patch** (1.0.1) |

Always run `npm run build` before publish (`prepublishOnly` handles this).

## 7. Maintenance plan

- [ ] Watch GitHub issues
- [ ] Re-test when new **Flutter stable** releases (platform templates change)
- [ ] Update `flutter_lints` / dependency versions in `generator.ts` periodically
- [ ] Document breaking changes in `CHANGELOG.md`

## 8. package.json snippets to add before publish

```json
{
  "repository": {
    "type": "git",
    "url": "https://github.com/YOUR_USERNAME/flutter-forge.git"
  },
  "homepage": "https://github.com/YOUR_USERNAME/flutter-forge#readme",
  "bugs": {
    "url": "https://github.com/YOUR_USERNAME/flutter-forge/issues"
  }
}
```

Replace `YOUR_USERNAME` with your GitHub username.

---

## 9. Common publish errors

| Error | Fix |
|-------|-----|
| Package name taken | Rename or use scoped `@yourscope/create-flutter-forge` |
| Missing files on install | Check `package.json` `"files"` includes `templates` |
| `prepublishOnly` failed | Run `npm run build` locally and fix TS errors |
| Wrong Node version | `engines.node` is `>=18`; document in README |

---

## 10. Scoped package alternative

If `create-flutter-forge` is taken:

```json
"name": "@salim1334/create-flutter-forge"
```

Users install with:

```bash
npm create @salim1334/flutter-forge@latest my_app
```

(Requires matching bin naming in docs.)
